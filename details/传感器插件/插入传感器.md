# 从 `launchsim.py` 启动 Gazebo 的运行路径（含 RGB 相机参数流）

## 1. 启动入口
你执行：

```bash
python3 src/cyberdog_simulator/cyberdog_gazebo/script/launchsim.py
```

`launchsim.py` 会按顺序启动 3 个脚本：
1. `launchgazebo.sh`（Gazebo 仿真）
2. `launchvisual.sh`（可视化）
3. `launchcontrol.sh`（控制）

其中第 1 步启动后会等待 5 秒，再启动后两步。

---

## 2. 调用链路（逐层）

### 第 0 层：`launchsim.py`
- 固定工程根目录：`/home/cyberdog_sim`
- 优先用 `gnome-terminal` 分终端启动；无图形环境就后台 `Popen`
- 关键脚本列表：
  - `src/cyberdog_simulator/cyberdog_gazebo/script/launchgazebo.sh`
  - `src/cyberdog_simulator/cyberdog_gazebo/script/launchvisual.sh`
  - `src/cyberdog_simulator/cyberdog_gazebo/script/launchcontrol.sh`

### 第 1 层：`launchgazebo.sh`
- `source /opt/ros/galactic/setup.bash`
- `source install/setup.bash`
- 实际执行：

```bash
python3 src/cyberdog_simulator/cyberdog_gazebo/script/gazebolauncher.py \
  ros2 launch cyberdog_gazebo race_gazebo.launch.py
```

### 第 2 层：`gazebolauncher.py`
- 只是包装器：接收后续命令并执行
- `Ctrl+C` 时会 `killall gzclient/gzserver`

### 第 3 层：`race_gazebo.launch.py`
主要做 3 件事：
1. 启动 `gzserver.launch.py`
2. （非 headless 时）启动 `gzclient.launch.py`
3. 调用 `/spawn_entity` 把机器人模型塞进 Gazebo

URDF 不是直接读 `.urdf`，而是：
- 找到 `robot.xacro`
- 通过 `xacro.process_file(..., mappings=...)` 生成 URDF 字符串
- 再通过 `spawn_entity` 注入仿真

---

## 3. 传感器参数是如何传到模型里的（RGB 为例）

在 `race_gazebo.launch.py` 中：
- 定义了 `use_rgb_camera` 启动参数（默认 `false`）
- 并传入 xacro 映射：`'USE_RGB_CAMERA': use_rgb_camera`

在 `robot.xacro` 中：
- 定义 `xacro:arg name="USE_RGB_CAMERA"`
- include `gazebo.xacro`

在 `gazebo.xacro` 中：
- 用 `<xacro:if value="$(arg USE_RGB_CAMERA)">` 包裹 RGB 相机传感器块
- 插件 `libgazebo_ros_camera.so` 发布图像与相机信息话题

即：

`ros2 launch 参数 -> race_gazebo.launch.py mappings -> robot.xacro 参数 -> gazebo.xacro 条件块 -> Gazebo 传感器插件`

---

## 4. 当前脚本体系下的注意点（很关键）

1. `launchsim.py` 走的是 `launchgazebo.sh`，而它默认没有传 `use_rgb_camera:=true`。  
   也就是说：即便你已经写好 RGB 代码，走 `launchsim.py` 默认仍不会开 RGB。

2. 要启用 RGB，有两种方式：
   - 方式 A（推荐临时验证）：绕过 `launchsim.py`，直接手动 launch：

```bash
source /opt/ros/galactic/setup.bash
source /home/cyberdog_sim/install/setup.bash
ros2 launch cyberdog_gazebo race_gazebo.launch.py use_rgb_camera:=true paused:=false headless:=True
```

   - 方式 B（长期使用）：修改 `launchgazebo.sh`，在命令后加上 `use_rgb_camera:=true`。

3. 若出现 `gzserver exit code 255`，通常是已有旧 `gzserver/gzclient` 占用。先清理再启动：

```bash
pkill -f "ros2 launch cyberdog_gazebo race_gazebo.launch.py" || true
pkill -f gzserver || true
pkill -f gzclient || true
```

---

## 5. 验证链路是否成功

新开终端执行：

```bash
source /opt/ros/galactic/setup.bash
source /home/cyberdog_sim/install/setup.bash
ros2 topic list | grep -E "rgb|camera|image"
ros2 topic hz /rgb/image_raw
ros2 topic echo /rgb/camera_info --once
```

若 `topic hz` 有频率输出，说明 RGB 相机已经在仿真中正常工作。

# 补充总结：Gazebo 中插入传感器插件通用流程（以 RGB 相机为例）

这份补充总结基于本次给 CyberDog 仿真加入 RGB 相机的完整过程整理，目的不是替换上面的原始记录，而是沉淀一套后续可复用的方法，方便继续添加深度相机、超声、IMU、雷达等其他传感器。

---

## 1. 这次 RGB 相机接入的整体结论

这次接入 RGB 相机，核心上做了 5 件事：

1. 在 `robot.xacro` 中准备传感器开关参数与安装位。
2. 在 `gazebo.xacro` 中增加 Gazebo 传感器块与 ROS 插件。
3. 在 `race_gazebo.launch.py` 中增加 launch 参数，并透传到 xacro。
4. 在启动脚本里默认打开该参数，让 `launchsim.py` 能真正带着相机启动。
5. 启动后用 ROS 话题、频率、显示工具逐步验证，并处理 GUI / QoS / install 未更新等问题。

一句话概括就是：

`launch 参数 -> xacro 参数 -> Gazebo 传感器插件 -> ROS 话题 -> 可视化验证`

---

## 2. RGB 相机这次具体改了哪些地方

### 2.1 `robot.xacro`
作用：定义参数、组织整机结构、包含 Gazebo 扩展。

本次关键点：
- 增加参数：`<xacro:arg name="USE_RGB_CAMERA" default="false" />`
- 让 `gazebo.xacro` 能读取这个参数
- 复用已有的 `RGB_camera_link` / `RGB_camera_joint` 安装位

经验：
- 如果传感器本体 `link/joint` 已经存在，就优先复用
- 如果不存在，就先在机器人主体上补安装位，再谈 Gazebo 插件

### 2.2 `gazebo.xacro`
作用：挂 Gazebo 专用扩展，包括摩擦、材质、传感器、插件等。

本次关键点：
- 用 `<xacro:if value="$(arg USE_RGB_CAMERA)">` 做条件启用
- 在 `RGB_camera_link` 上挂 `type="camera"` 的 Gazebo 传感器
- 插件使用 `libgazebo_ros_camera.so`
- 通过 `<ros><remapping>...</remapping></ros>` 导出 ROS 话题
- 通过 `<frame_name>` 指定 TF 坐标系

通用原则：
- `robot.xacro` 管“模型结构”
- `gazebo.xacro` 管“仿真行为和插件”

### 2.3 `race_gazebo.launch.py`
作用：把 launch 参数传给 xacro。

本次关键点：
- 增加 `use_rgb_camera` 参数
- 在 `xacro.process_file(..., mappings=...)` 中加入：

```python
'USE_RGB_CAMERA': use_rgb_camera
```

如果只改了 `xacro`，没改这里，传感器不会真正打开。

### 2.4 `launchgazebo.sh`
作用：给日常启动入口补默认参数。

本次关键点：
- 让 `launchsim.py` 间接启动 Gazebo 时，也能默认启用 RGB 相机
- 否则会出现“手工 `ros2 launch` 能看到相机，`launchsim.py` 启动却没有相机”的现象

### 2.5 `launchsim.py` / `launchcamera.sh`
作用：自动打开相机显示窗口。

本次关键点：
- 在 `launchsim.py` 启动流程中纳入 `launchcamera.sh`
- `launchcamera.sh` 负责：
  1. 自动探测图像话题
  2. 处理 QoS 不兼容问题
  3. 再调用显示工具打开窗口

## 3. 后面新增其他传感器时，建议按这个流程做

### 3.1 第一步：先确认“装在哪里”
先确认传感器挂载位置：
- 是否已有对应 `link` / `joint`
- 坐标系朝向是否合理
- 是否需要新建安装位

若是相机，重点关注：
- `xyz`
- `rpy`
- 视场方向是否朝前

若姿态不对，经常会出现“有图像，但画面是地面、天空或一片灰”的现象。

### 3.2 第二步：在 `robot.xacro` 里加参数开关
建议每个可选传感器都做成独立开关，例如：
- `USE_RGB_CAMERA`
- `USE_DEPTH_CAMERA`
- `USE_ULTRASONIC`
- `USE_EXTRA_LIDAR`

这样后续调试时可以按需启停，避免所有传感器一次性全部加载。

### 3.3 第三步：在 `gazebo.xacro` 里加传感器块
一般要明确以下内容：
- `reference` 挂在哪个 `link` 上
- 传感器类型：`camera`、`ray`、`gpu_ray`、`imu` 等
- 刷新率：`update_rate`
- 插件文件名
- ROS 话题 remapping
- `frame_name`

优先检查这几项：
- 插件 `.so` 名字对不对
- `reference` link 是否存在
- 话题 remapping 是否与预期一致
- `frame_name` 是否与 link 对应

### 3.4 第四步：在 launch 层把参数透传下去
很多“代码写了但没效果”的问题，都出在这里。

必须检查：
1. `DeclareLaunchArgument(...)` 是否新增了参数
2. `LaunchConfiguration(...)` 是否正确读取
3. `xacro.process_file(..., mappings=...)` 是否把参数传入
4. 启动脚本是否真的附带了该参数

### 3.5 第五步：重新编译并刷新环境
这一步非常关键，因为运行时读的通常不是 `src`，而是 `install/share`。

推荐命令：

```bash
cd /home/cyberdog_sim
colcon build --packages-select cyberdog_description cyberdog_gazebo --merge-install 
source /opt/ros/galactic/setup.bash
source /home/cyberdog_sim/install/setup.bash
```

如果你只改了 `src` 但没重新 `build + source`，很容易误以为“插件没生效”。

### 3.6 第六步：先验证话题，再验证显示
不要一上来只盯着 GUI 窗口，建议按下面顺序排查：

1. Gazebo 是否成功启动
2. 传感器相关 topic 是否出现
3. topic 是否有频率
4. topic 内容是否在变化
5. 最后再看图形显示是否正常

## 5. 这次遇到的关键问题与对应经验

### 5.1 修改了源码，但启动结果没变化
原因通常是：
- 没重新编译
- 没重新 `source install/setup.bash`
- 启动读的是 `install/share`，不是 `src`

处理方式：
- `colcon build`
- 重新 `source`
- 再次启动验证

### 5.2 `gzserver exit code 255`
原因通常是旧进程没退出，端口或资源被占用。

处理方式：
- 先 `pkill` 清理旧的 `gzserver` / `gzclient`
- 再重新 launch

### 5.3 GUI 打不开或 `gzclient` 崩溃
这次出现过 X11 / 权限相关问题。

经验：
- 先用 `headless:=True` 验证传感器链路
- GUI 相关问题和传感器逻辑分开排查
- 不要把“GUI 打不开”误判为“插件没生效”

### 5.4 相机话题有了，但显示窗口空白
这次的根因是 QoS 不兼容。

Gazebo 相机插件发布端常见是：
- `BEST_EFFORT`

而一些显示工具默认订阅使用：
- `RELIABLE`

两边不兼容时，就会出现：
- 明明 `topic hz` 有数据
- 但 `image_view` / `rqt_image_view` 看不到图像

所以以后遇到“有 topic、也有频率，但窗口空白”，优先检查 QoS。

### 5.5 `camera_info` 不是图像
本次也确认了几个常见话题的用途：
- `/rgb_camera_sensor/image_raw`：真正的图像
- `/rgb_camera_sensor/camera_info`：相机内参，不是画面
- `/rgb_camera_sensor/image_raw_mouse_left`：交互事件，不是图像主体

以后排查时，不要选错 topic。

### 5.6 画面发灰、像没内容
不一定是没数据，可能只是：
- 视角方向不对
- 机器狗没站起来
- 相机姿态朝向地面、天空或模型内部

所以当图像“看起来不对”时，要同时检查：
- 传感器 pose
- 机器人实际姿态
- 场景里是否有可见物体

## 6. 关于显示工具的经验总结

本次排障中，显示工具经历了几轮调整：
- 直接用 `image_view`
- 改用 `rqt_image_view`
- 尝试 `image_tools/showimage`
- 最终采用“QoS 中继 + `rqt_image_view`”

最终方案之所以更稳，是因为它把问题拆成两层：
1. 先解决订阅 QoS 与 Gazebo 发布端不兼容的问题
2. 再用更方便缩放窗口的 GUI 工具显示

这个思路以后对其他传感器也适用：
- 先确认数据链路通了
- 再考虑显示工具是否适配
- 工具不兼容时，可以加中继节点做协议/QoS 适配

## 7. 以后新增其他传感器插件时，建议重点检查这些点

### 结构层
- 安装位 `link/joint` 是否存在
- 坐标系是否合理
- 名称是否统一

### xacro 层
- 是否定义了开关参数
- 是否被 include 到最终模型
- 条件块是否真的能命中

### launch 层
- 参数是否声明
- 参数是否透传到 xacro mappings
- 启动脚本是否真的带上该参数

### 运行层
- 是否重新编译并 `source`
- 是否有旧进程残留
- 是否读到了最新 install

### ROS 层
- topic 是否出现
- topic 名字是否和 remapping 一致
- `frame_name` 是否合理
- QoS 是否兼容

### 显示层
- 订阅的是不是正确 topic
- 工具是否支持当前 QoS
- 窗口空白到底是“无数据”还是“订阅失败”

## 9. 最后给后续加新传感器的建议

以后每加一个新传感器，建议都按下面的节奏推进：

1. 先确认安装位和坐标系。
2. 再写 `gazebo.xacro` 里的插件块。
3. 然后补 launch 参数透传。
4. 编译并重新 `source`。
5. 先用 `ros2 topic list / hz / echo / info -v` 验证数据链路。
6. 最后再处理 GUI 显示、窗口大小、交互体验等问题。

这样做的好处是：
- 每一步都能单独验证
- 出问题时能快速定位在“模型层 / launch 层 / Gazebo 层 / ROS 层 / GUI 层”中的哪一层
- 不会把多个问题混在一起排查

如果后面要加其他传感器插件，可以直接参考这里，把“RGB 相机”替换成对应的传感器类型即可。
